# Hermes Freshness Report-Only Integration

Task: `TASK-HERMES-FRESHNESS-REPORT-ONLY-INTEGRATION-001`

Status: implemented locally

Date: 2026-06-13

## Verdict

`official-sources hermes scheduled-freshness-report` is a report-only wrapper around the existing
freshness report engine. It prepares a deterministic report path, runs
`official-sources hermes freshness-report`, and returns the subprocess result.

```text
operational role: report generation only
timer activation: forbidden
VPS deployment: forbidden
strict-audit gate change: forbidden
repair role: forbidden
```

## Command Shape

The wrapper runs from the repository/runtime root and passes `--runtime-root .` to the underlying
freshness command.

```bash
official-sources hermes scheduled-freshness-report \
  --repo-root /opt/official-sources/app \
  --state-root /var/lib/hermes-official-sources-auditor
```

The generated freshness command uses:

```text
--runtime-root .
--output <state-root>/freshness-reports/hermes-freshness-report-YYYYMMDD-HHMMSS.md
```

Default expected and critical sources:

```text
BOE
BDNS
BOCM
```

Default threshold:

```text
72 hours
```

## Exit-Code Policy

The integration does not parse or enforce `VERDICT: NO-GO`.

```text
report generated by freshness-report: return freshness-report exit code
freshness-report returns 0 with VERDICT NO-GO: wrapper returns 0
freshness-report cannot read runtime state or write report: wrapper returns non-zero
missing official-sources executable: wrapper returns 127
```

The failure boundary is operational generation failure, not source staleness.

## Write Surface

Allowed write:

```text
<state-root>/freshness-reports/hermes-freshness-report-YYYYMMDD-HHMMSS.md
```

Forbidden:

```text
- systemd timers or services
- VPS deployment
- git pull
- ingest-monitor-date
- SQLite writes
- config/sources.yaml mutation
- official_documents
- evidence
- candidates
- publication artifacts
- downstream smokes or writes
- automatic issues
- strict-audit enforcement changes
```

## Retention

Expected retention remains the policy defined in
`docs/architecture/hermes-freshness-schedule-policy.md`: keep the latest 60 freshness reports under
`<state-root>/freshness-reports/`.

This PR does not implement retention deletion because that would broaden the write surface beyond
single-report generation.
